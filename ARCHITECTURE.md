# Legio Architecture

> Multi-agent orchestration for Claude Code. Version 0.1.0.

## System Overview

Legio turns a single Claude Code session into a coordinated multi-agent team. It spawns worker agents as Claude Code processes inside git worktrees, coordinates them via a SQLite mail system, and merges their branches back with tiered conflict resolution.

**The Claude Code session that runs `legio` IS the orchestrator.** There is no separate daemon process. Hooks + CLI + SQLite provide everything.

### Key Design Decisions

| Decision | Rationale |
|---|---|
| **SQLite mail over beads** | ~1–5ms per query vs. beads file I/O; WAL mode enables concurrent multi-agent access |
| **Git worktrees for isolation** | Each agent gets its own working tree on a dedicated branch; merging is explicit |
| **Tmux sessions for agents** | Claude Code runs as a subprocess in a named tmux session; enables attach, capture-pane, kill |
| **No runtime npm dependencies** | `better-sqlite3` is the only native dep; all external tools invoked via `node:child_process` spawn |
| **TypeScript with strict mode** | `noUncheckedIndexedAccess` + `noExplicitAny` enforced; all types in `src/types.ts` |

### Hook Lifecycle

Claude Code hooks wire legio into every agent session automatically:

```
SessionStart   → legio prime --agent <name>       # Load config + mulch expertise
UserPromptSubmit → legio mail check --inject       # Surface new messages from agents
PreToolUse     → block git push, AskUserQuestion,  # Safety guards
                 EnterPlanMode, EnterWorktree
PostToolUse    → legio log tool-end + mail check   # Record events, deliver mail
Stop           → legio log session-end + mulch learn # Finalize session + capture learnings
PreCompact     → legio prime --compact             # Re-inject context after compaction
```

## Agent Hierarchy

### The 10 Agent Types

| Agent | Depth | Can Spawn | Role |
|---|---|---|---|
| **coordinator** | 0 | leads only | Strategic orchestrator; decomposes objectives, dispatches leads, merges work |
| **gateway** | 0 | none | Planning companion; analyzes codebase, creates beads issues — no file writes, no spawning |
| **cto** | 0 | none | Strategic analyst; reads codebase + metrics, writes `strategy.json` recommendations |
| **supervisor** | 1 | builders, scouts, reviewers | Per-project manager; owns a work stream end-to-end |
| **lead** | 1 | builders, scouts, reviewers, mergers | Team lead; explores, specs, implements, reviews within a feature area |
| **monitor** | 1 | none | Tier 2 continuous fleet patrol; runs at root, no worktree |
| **builder** | 2 | none | Leaf; writes code, runs tests, commits to worktree branch |
| **scout** | 2 | none | Leaf; read-only exploration + spec writing via `legio spec write` |
| **reviewer** | 2 | none | Leaf; read-only validation, reports pass/fail via mail |
| **merger** | 2 | none | Leaf; resolves merge conflicts using 4-tier pipeline |

### Hierarchy Enforcement

Spawning rules are code-enforced in `src/commands/sling.ts`:
- Coordinators can only spawn leads (depth 0 → 1)
- Leads/supervisors can only spawn depth-2 agents (depth 1 → 2)
- Depth-2 agents cannot spawn anything (`canSpawn: false` in manifest)
- `--force-hierarchy` bypasses validation (debugging only)

Max depth is configurable in `config.yaml` (`agents.maxDepth`, default 2).

### Two-Layer Agent Configuration

Each agent session gets two instruction layers:

1. **Base definition** (`agents/*.md`) — the HOW: workflow, constraints, tool access, failure modes
2. **Dynamic overlay** (`.claude/CLAUDE.md` in the worktree) — the WHAT: task ID, spec path, file scope, branch, parent agent

The overlay is generated by `legio sling` via `src/agents/overlay.ts` using `templates/overlay.md.tmpl`.

## Messaging System

### SQLite Mail (`mail.db`)

All inter-agent communication flows through `src/mail/store.ts`, backed by SQLite with WAL mode:

```sql
messages (
  id TEXT PRIMARY KEY,          -- "msg-" + nanoid(12)
  from_agent TEXT,
  to_agent TEXT,
  subject TEXT,
  body TEXT,
  type TEXT,                    -- semantic or protocol type
  audience TEXT,                -- 'human' | 'agent' | 'both'
  priority TEXT,                -- 'low' | 'normal' | 'high' | 'urgent'
  thread_id TEXT,               -- conversation threading
  payload TEXT,                 -- JSON-encoded structured data
  read INTEGER,
  created_at TEXT
)
```

### Message Types

**Semantic types** (human-readable status updates):
- `status` — progress update
- `question` — needs clarification from parent
- `result` — task output
- `error` — something broke

**Protocol types** (structured coordination signals):
- `worker_done` — leaf signals task complete (triggers merge pipeline)
- `merge_ready` — supervisor confirms branch is verified
- `merged` / `merge_failed` — merger reports merge outcome
- `escalation` — any agent escalates an issue upward
- `health_check` — watchdog probes agent liveness
- `dispatch` — coordinator assigns a task to an agent
- `assign` — general task assignment

### Group Addressing

`src/mail/broadcast.ts` resolves group addresses to individual agent names:
- `@all` — all active agents except sender
- `@builders`, `@scouts`, `@reviewers`, `@leads`, `@mergers`, `@supervisors` — by capability
- Singular and plural forms both accepted (`@builder` = `@builders`)

### Mail Injection

The `--inject` flag on `legio mail check` writes unread messages as a system-reminder into the Claude context. The `PostToolUse` hook calls this automatically after every tool call (with 500ms debounce), ensuring agents receive messages without polling.

## Merge Pipeline

### FIFO Merge Queue (`merge-queue.db`)

`src/merge/queue.ts` implements a SQLite-backed FIFO queue:

```sql
merge_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,  -- guarantees FIFO ordering
  branch_name TEXT,
  bead_id TEXT,
  agent_name TEXT,
  files_modified TEXT,                   -- JSON array
  enqueued_at TEXT,
  status TEXT,                           -- pending | merging | merged | conflict | failed
  resolved_tier TEXT                     -- which tier resolved it
)
```

### 4-Tier Conflict Resolution

`src/merge/resolver.ts` attempts tiers in sequence, stopping at first success:

| Tier | Name | Strategy |
|---|---|---|
| **Tier 1** | `clean-merge` | Standard `git merge`; no conflicts → done |
| **Tier 2** | `auto-resolve` | Parse conflict markers; keep incoming (agent) changes |
| **Tier 3** | `ai-resolve` | Use Claude to reason about conflicts and produce resolution |
| **Tier 4** | `reimagine` | Abort merge; reimplement changes from scratch in a fresh branch |

Disabled tiers are skipped. Config flags: `merge.aiResolveEnabled`, `merge.reimagineEnabled`.

JSONL files get special handling: union-dedup-sort by `id` + `recorded_at` instead of generic conflict parsing.

### Merge Flow

```
builder sends worker_done mail
  → supervisor verifies branch (quality gates, file diff)
  → supervisor enqueues branch in merge-queue.db
  → supervisor sends merge_ready to coordinator
  → coordinator runs legio merge --branch <name>
  → resolver attempts Tier 1 → 2 → 3 → 4
  → merger sends merged/merge_failed mail
```

## Observability Stack

### The 5 SQLite Databases

| Database | Location | Contents |
|---|---|---|
| `mail.db` | `.legio/mail.db` | Inter-agent messages (WAL, busy_timeout=5000) |
| `sessions.db` | `.legio/sessions.db` | Agent sessions + coordinator runs |
| `events.db` | `.legio/events.db` | Tool invocations, session lifecycle, errors |
| `metrics.db` | `.legio/metrics.db` | Session metrics, token usage, cost snapshots |
| `merge-queue.db` | `.legio/merge-queue.db` | FIFO merge queue with resolution status |

All databases use `PRAGMA journal_mode=WAL` and `PRAGMA busy_timeout=5000`.

### EventStore Schema (`events.db`)

```sql
events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id TEXT,
  agent_name TEXT NOT NULL,
  session_id TEXT,
  event_type TEXT NOT NULL,    -- tool_start | tool_end | session_start | session_end | mail | error
  tool_name TEXT,
  tool_args TEXT,              -- filtered (secrets redacted, large args truncated)
  tool_duration_ms INTEGER,
  level TEXT,                  -- debug | info | warn | error
  data TEXT,                   -- JSON extra context
  created_at TEXT
)
```

Indexed on: `(agent_name, created_at)`, `(run_id, created_at)`, `(event_type, created_at)`, `(tool_name, agent_name)`, `level` (partial index on `error`).

### SessionStore Schema (`sessions.db`)

Tracks `AgentSession` rows (state machine: `booting → working → completed | stalled | zombie`) plus `runs` table grouping sessions by coordinator run ID.

### MetricsStore Schema (`metrics.db`)

Per-session token counts (input, output, cache read, cache creation) + cost estimates. Also tracks live `token_snapshots` for the `legio costs --live` command.

### Run Groupings

`legio run` groups agent sessions under a coordinator run ID (stored in `.legio/current-run.txt`). Enables cost attribution and replay filtering across a single work session.

## Watchdog System

### 4-Tier Architecture

| Tier | Name | Mechanism |
|---|---|---|
| **Tier 0** | Mechanical daemon | `legio watch` — polls tmux + pid liveness on interval (default 30s) |
| **Tier 1** | AI triage | Ephemeral Claude session analyzes stalled agent; classifies failure |
| **Tier 2** | Monitor agent | Persistent Claude session (`legio monitor`) — continuous fleet patrol |
| **Tier 3** | Supervisor | Per-project supervisor agents monitor their own sub-workers |

### Progressive Escalation (Tier 0)

For each stalled agent, escalation levels increase on successive checks:

```
Level 0 (warn)     → log warning, no action
Level 1 (nudge)    → send tmux nudge via legio nudge
Level 2 (escalate) → invoke Tier 1 AI triage (if enabled)
Level 3 (terminate) → kill tmux session
```

`nudgeIntervalMs` (default 60s) controls time between escalation stages.

## Quality Gates

### Test Framework

Vitest with unit/integration split (`vitest.workspace.ts`):
- **Unit:** excludes 7 slow integration test files
- **Integration:** coordinator, sessions store, watchdog, and other I/O-heavy tests

Tests are colocated with source files (e.g., `src/commands/init.test.ts`).

### Philosophy

- Real implementations over mocks: temp dirs for filesystem, `:memory:` or temp files for SQLite, real git repos for worktree tests
- Mock only when real thing has unacceptable side effects: tmux, AI API calls, network requests

### Agent Quality Gates (before completion)

```bash
bun test              # All tests must pass
bun run lint          # Biome: format + lint (tab indentation, 100-char line width)
bun run typecheck     # tsc --noEmit (strict mode, noUncheckedIndexedAccess, noExplicitAny)
```

Agents must pass all three before running `bd close` or sending `worker_done` mail.

### PreToolUse Safety Hooks

Deployed to every agent worktree via `src/agents/hooks-deployer.ts`:
- **`git push` block** — all push blocked everywhere; `legio merge` handles integration
- **`AskUserQuestion` block** — agents use `legio mail send --type question` instead
- **`EnterPlanMode` block** — agents execute immediately (propulsion principle)
- **`EnterWorktree` block** — agents already have assigned worktrees; nesting disallowed
